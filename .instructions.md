# AI Assistant Instructions — Design Token Forge

## Project Identity

This is **Design Token Forge** — a design system generator platform. It provides a unified token system built on the **Tokn 4-tier color architecture** (T0 primitives → T1 semantic roles → T2 surface contexts → T3 component sizes) with ~3,000+ CSS custom properties and ~35 L1 components where every visual property is a variable.

## Core Principles (NEVER violate these)

1. **Zero hardcoded values in components.** Every visual property must be a CSS custom property. If you write a color, size, spacing, radius, shadow, or any visual value directly — it's wrong. Wrap it in a variable.

2. **Layered resolution.** Variables resolve through 5 layers: Defaults → Global Tokens → Product Tokens → Component Tokens → Instance. Always respect this cascade.

3. **Variable naming convention:**
   - Tokn T0 primitives: `--prim-{palette}-{step}` (e.g., `--prim-brand-500`, `--prim-danger-Color600`)
   - Tokn T1 semantic: `--{role}-{group}-{property}` (e.g., `--primary-component-bg-default`)
   - Tokn T2 surface: `--{surface}-{group}-{property}` (e.g., `--surface-base-ct-default`)
   - DTF global extras: `--{category}-{name}-{scale}` (e.g., `--radius-md`, `--shadow-lg`)
   - Component tokens: `--{component}-{property}` or `--{component}-{slot}-{property}` (e.g., `--btn-radius-tl`, `--btn-icon-gap`)
   - Variant/state tokens: `--{component}-{variant}-{property}-{state}` (e.g., `--btn-primary-bg-hover`)

4. **Framework-agnostic.** The core is CSS. JS wrappers are thin prop-to-class mappers + a11y attributes. Never put visual logic in JS — it belongs in CSS variables.

5. **Shape is not just border-radius.** Shape encompasses: radius (4 corners), border-width (4 sides), border-style, shadow, ring, outline, clip-path, overflow. Always expose the full shape surface.

6. **Every component has 7 axes:** Shape, Dimension, Surface, Typography, Slots, Motion, A11y. Don't skip axes.

## Key Architecture Facts

- **Monorepo:** pnpm workspaces at `/packages/*`
- **Three packages:** `@design-token-forge/tokens`, `@design-token-forge/components`, `@design-token-forge/generator`
- **Specs live in:** `/specs/tokens/` (YAML token schemas), `/specs/components/` (YAML component variable schemas)
- **Docs live in:** `/docs/` (architecture, tokens, components, decisions)
- **Variable count:** ~3,000-3,500 total (~775 global [Tokn + DTF extras] + ~2,200-2,700 component-level)
- **Component count:** ~35 L1 components
- **Density modes:** 10 — micro, tiny, small, base, medium, large, big, huge, mega, ultra
- **Color architecture:** Tokn 4-tier (T0 primitives → T1 roles → T2 surfaces → T3 sizes)
- **Surface contexts:** 9 — surface-base, surface-bright, surface-deep, surface-accent, surface-dim, container, over-container, float, inverse
- **Semantic roles:** 6 — primary, brand, success, warning, danger, info
- **Color scheme:** light/dark (via Tokn T1/T2 dual-mode tokens)

## Variable Axis Reference

When defining or reviewing component variables, ensure all 7 axes are covered:

```
📐 Shape:      radius (4 corners), border (4 sides × width/style/color), shadow, ring, outline, clip-path, overflow
📏 Dimension:  height × size, padding x/y × size, min/max, gap, width-model
🎨 Surface:    bg × variant × state, fg × variant × state, opacity, backdrop
✏️ Typography: family, size × scale, weight, line-height, letter-spacing, transform
🧩 Slots:      each sub-element gets its own size, color, gap, typography, shape subset
⚡ Motion:     transition property/duration/easing, enter/exit animations
♿ A11y:       focus-visible outline, min-tap-target, forced-colors
```

## Component Variable Schema Convention

Component specs in `/specs/components/` follow this structure:

```yaml
component: button
variables:
  shape:
    - name: --btn-radius-tl
      default: var(--radius-md)
      description: Top-left border radius
    # ... all shape vars
  dimension:
    # ... all dimension vars
  surface:
    # ... by variant × state
  typography:
    # ...
  slots:
    icon:
      # ... icon sub-element vars
    loader:
      # ... loader sub-element vars
  motion:
    # ...
  a11y:
    # ...
```

## What NOT to Do

- Don't create components with hardcoded colors, sizes, or spacing
- Don't add framework-specific code to `/packages/components/` core — only in framework-specific wrapper subdirectories
- Don't skip the slot axis — sub-elements need their own variables
- Don't use `!important` in component CSS
- Don't assume only light/dark themes — theme dimensions are orthogonal (color-scheme × density × motion × contrast × brand)
- Don't invent new color tokens — use Tokn T0/T1/T2 semantic tokens. Surface and role tokens exist for every context.
- Don't use the old size names (xs/sm/md/lg/xl) — use density mode names (micro/tiny/small/base/medium/large/big/huge/mega/ultra)
- Don't create L2 components (layouts, page patterns) — that's out of scope. Only L1 primitives.
- Don't create new documentation or summary files unless explicitly requested by the user
